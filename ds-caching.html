Cacheing

  • the only application-transparent technique to reduce end-to-end latency
  • application-transparent: don’t have to touch application
  • when don’t use: a situation in which the hit rate would be low because data is typically only accessed once, such as for streaming video
  • when use:
    • high cache advantage: hit and read from local
    • cost involved besides time: you have to pay for the bandwidth for accessing the remote copy, but accessing a cached copy is free

Basic Concept

  • Bare-bones idea
    • You need to make multiple references to an object
    • The object is far away
    • You make a local copy and access it instead
    • Do all this transparently to programs and users (“under the covers”)
  • Local storage used for copies is referred to as “the cache”
  • Local copy of a single object is called “a cache copy”
  • Transparency is key: completely invisible (except better performance)

Simple Cache Metrics

  • references → number of attempts to find an object in the cache
  • HIts → number of successes
  • Misses → nmber of failures
  • Miss Ratio = Misses/Reference
  • Hit Ratio = Hits/Reference = 1 - Miss Ratio
  • Expected cost of a reference = (miss ratio * cost of miss) + (hit ratio * cost of hit)
    • expected cost of a reference < cost of a miss → cache improve performance
  • Cache Advantage = (Cost of Miss / Cost of Hit)
      • (where cost is measured in time delay to access object)

Double Win

  • a twofor not a tradeoff
  • cacheing stie obiously wins big
    • local copy → lowerst possible latency
    • many cources of end-to-end latency avoided
      • no network queuing delays
      • no server queuing or computation delays
      • no tranmission delays
  • Rest of the distributed system also wins big
    • Caching lowers utilization of shared resources
    • less network traffic, less server load
    • your caching makes life better for everyone

Deep Assumption

  • multiple refereces → caching suseless for just one reference
    • on any reference i how do you know there will be more?
    • typical assumption: one reference → others likely
    • empirical observation about real systems in real use
    • temporal locality of reference/temporarl locality
    • assumption somtimes fails to be true → caaching wastful

Wide Application

Questions

  • Fetch Policy → what date should you cache and when
  • Update Propagation Policy → How do updates become visible everywhere
  • Cache Replacement Policy → What old data do you throw out to free up space
  • if you have infinite cache space, #3 is a non-issue
    • but old cache misses still cause delay
    • so prefetching becomes very important

Fetch Policy

approach 1: Full Replication

  • propoerties:
    • every participating machine gets a full and complete copy
    • every new file gets tramsmitted to all replicas
    • every updated file gets propagated: no well-defineed semantics for when updates are propagated
    • All data is fetched in advance to point of use
  • sync solutions
  • Shortcomings
    • High Space Using: Storage for entire subtree consumed on every replica
    • significant update traffic on hot spots:
      • painful on metered networks
      • no well-defined semantics for when you see updates
    • Machines receive updates whether they care or not
      • push model of update propagating
      • you get whole subtree regardless if you need them or not
    • coarse-grain, non-selective management of data

approach 2: On-demand Caching

  • tranparently fetch file only if needed
    • approach used in AFS (inherited and extended from AFS-2 by Coda File System)
    • requires integration with the operation system
    • fine-grained and selective approach to data management
  • shortcoming:
    • requires operation system modifications
  • benefits:
    • total application transparency
    • enable demand caching

Caching in the Real World

  • reality1: cost of remote data access of then not uniform
    • often takes the form Ax + B for x bytes
    • may be mroe plex as x gets larger (e.g. TCP file transfer)
  • reality2: Nearby objects often accessed soon after object access
    • another empirical observation about real systems in real use
    • refered to as “spatial locality” of reference” or “spatial locality”
  • reality3: remote data more coaresly addressable than local
    • typically a scalability tradeoff at next level of memory hierachy. Same number of address bits can cover large volume of data.
  • Combining observation → fetch more than you need on miss
    • effectively amortizes cost of fetch
    • addumption somtimes violated
    • extra data fetched is the nuseless (may even hurt) can be viewed as crude form of prefetching

Spatial and temporal locality

  • Temporal and spatial locality are very differnet properties
    • caching implementations often tightly combine these assumptions
    • one can exist without the other
      • spactial without temporal: linear scan of huge file
      • temporal without spatial: tight loop accessing just one project

Update Propagation Policy (cache consistency)

one-copy semantics (safety property)

  • perfect functional transparency
  • caching system has one-copy semantics iff
    • there are no externally observable functional differences
    • relative to same system without caching (or replication)
    • all clients see contents of file identically as if only one copy of file existed
    • if caching is used: after an update operation, no program can observe a discrepancy between data in cache and stored data
  • No pattern of interleaved inputs to different nodes of the system can produce reliably distinguishable outputs in the two cases
  • Too difficult to achieve in practice. Except in very narrow circumstances
  • Goodness criteria for implementation
    • how little dafety can you give up. yert remain scalable
    • how complex is the implementation
  • how to approximate? performance tradeoff?

Hardness of one-copy semantics

  • physical master copy may not sexist
    • hosts track who has most recent copy
    • more likely in a P2P scenarios than client-server scenarios
    • also common in multiprocessor hardware caches
  • Network may break between some user and master copy
    • disconnected sites see no further updates
    • other sites don’t see updates by diconnected site
  • Intense read- and write-sharing across sites
    • generates huge amount of cache propagation traffic
    • interconnect becomes bottleneck for cache access
    • neither writer-bias (write-back) nor reader-bias (write-through) helps
    • caches effectively useless

Cache consistant strategies

broadcast invalidations

  • Every potential caching site notified on every update
    • no check to verify caching site actually contains object
    • notification includes specific object being invalidated
    • effectively broadcast of address being modified
  • At each site, next reference to object will cause a miss
  • problem: increasingly slow as the scale increase
    • network: packet transmission is not perfect. bus: transmission is perfect, as soon as request send, it will be delivered except hardware failure.
    • works well for modest site hardware application
  • strengths
    • very strict emulation of one-copy semantics
    • no race conditions if updater blocked until all caches invalidated
    • simple to implement
  • limitations
    • wasted traffic if no reader elsewhere
    • updating process blocked until invalidation complete
    • not a scalable design

check on use

  • reader checks master copy before each use
    • conditional fetch, if cache copy stale
  • has to be done at coarse granularity (entire file or large block)
    • otherwise every read is slowed down excessively
  • check validation before open(), do everything locally, and check only after close()
  • at whole file granularity → “session semantics”
    • open {write | read} close → session
    • approximation to strict one-copy semantics at bit-granularity
  • advantages:
    • strict consistency at coarse granularity
    • easy to implement, no server state
    • server don’t need to know of caching sites
    • to have better performance:
      • file size needs substantial
      • needs relatively long read-close session
  • disadvantages
    • slows read access on loaded servers & high-latency networks
    • check is almost always success → frivolous traffic
    • load on network and server
  • up to date is relative to network latency

Callback

  • targeted notification of caching sites
  • master copy tracks sites with cached data
  • typically done at coarse granularity (entire file). Can be made to work with byte ranges.
  • on update, all sites with cached data notified (callback)
  • Advantage
    • excellent scalability for unix workloads
    • zero network traffic for read of cached-valid objects
    • precursor to caching for disconnected operation
    • biases read performance in favor of write-performance
  • Disadvantage
    • sizable state on server
    • complexity of tracking cached state on clients
    • silence ambiguous for client
      • network failure → lost callbacks
      • periodic “keepalive” probes
      • data could be stable between probes
    • Nat networks with masquerading firewalls

Lease

  • basic idea
    • caching site obtains finite-duration control from master copy
    • duration: lease period → typically few seconds
    • multiple sites can obtain read lease; only one can get write lease
    • lease have to be renewed, else control expires with lease
    • multiple leases or at most one write leases at a time
  • Limiting cases
    • duration = 0 → check on use
    • duration = infinite → callback
  • Advantage
    • generalized the check on use and callback schemes
    • lease duration can be tuned to adapt to mutation rate
    • conceptually simple yet flexible
  • Key assumption
    • clocks tick at the same rate everywhere
    • absolute time doesn’t matter
    • only relative time matters
    • time becomes a hidden communication channel
  • disadvantage
    • lease-holder has total autonomy during lease; revocation
    • writers delayed while read lease holder complete their lease
    • more traffic than call back (but less than check on use): keepalives for callback only one per server, not per lease

skip scary parts

  • basic idea
    • when write-sharing detected, turn off caching on every client
      • all references go directly master copy
    • resume caching when write-sharing ends
  • advantage
    • precise single-copy semantics
    • excellent fallback position
    • good adaptation of caching aggressiveness to workload
  • disadvantage
    • server maintains state
    • server aware of every use of data

Faith-based caching (eventual consistency)

  • basic idea
    • blindly assume cached data is valid for a while
    • periodically check (based on time since last check)
    • no communication needed during trust period
  • imprecise and weak approximation to one-copy semantics
    • not just in the presence of network failures (partition)
    • no clear basis for reasoning about state of system
  • advantage
    • simple implementation
    • stateless server
  • disadvantage
    • user0visible inconsistencies sometimes seen
    • blind faith sometimes misplaced
    • not as efficient as callback-based approach

Pass the Buck

  • basic idea
    • let the user trigger cache revalidation (hit reload)
    • otherwise, all cached copies assumed valid forever
    • equivalent to infinite-TTL faith-based caching
    • across in the context of the web
  • advantages
    • trivial to implement, no server changes
    • avoids frivolous cache maintenance traffic
  • disadvantages
    • places burden on user
    • assumes existence of user: pain for write scripts/programs
  • Easier to reason about than faith-based caching
    • until you hit “reload”

Cache replace policy

picking a victim

  • may involve many factors
    • how soon evicted object might be needed again
    • how expensive it might be to fetch again
    • how much space is freed up by eviction
  • ideal victim: every large object that is not needed for a long time, and it very check to fetch again. In the face of failures, pain of non-serviceable miss also relevant.
  • Uniform fetch cost assumption reasonable near hardware
  • Non-uniform cost more likely at higher system layers
    • variable size objects are obvious source of non-uniform cost’
    • rotational/seek delay in disks is another source
    • differing network quality to different servers
  • vast majority of research assumes:
    • fixed size, fixed fetch cost, equal importance
    • only significant variable is then distance in the future to next reference to an object
    • considerable implementation
  • Abstract problem formulation
    • set of equal-sized data containers (aka “frames” in VM mgmt)
    • large set of equal-sized, equal-importance data objects (aka “pages”)
    • sequence of integers(“reference string”) → ids of accessed pages
    • only metric of interest is miss ratio

Predicting Distance to Next Reference

  • If reference string known in advance, problem is simple
    • assume OPT is the (theoretical) optimal replacement algorithm
    • it picks object that will be referenced furthest in future
    • challenge is to approximate OPT as closely as possible
  • Nothing approximates OPT closely over all reference strings
    • LRU is often a good approximation to OPT (not always)
    • assumes recent past is a good predictor of the near future, captures temporal locality very well & induces total ordering of frames

Working Site Concept

  • Given a time interval T, WorkingSet is the set of distinct data objects accessed in T
    • often, size of this set is all that matters (working set size)
    • function of width of interval, T, and its position relative to start
  • this captures adequacy of cache size relative to program behavior
  • for any given interval T
    • small working set → small cache is enough → high locality
    • large workin set → poor locality
    • size and pages in working set may change over time

When is LRU Ineffective

  • purely sequential access (aka “scan”) → no locality at all! caching cannot help at all; only adds overhead
  • purely random access → being smart is useless. ratio of cache size to total data size is all that matters
  • Real multi-process workloads can be mixtures of individual behavior
    • some process/threads may show locality
    • others may show random or sequential behavior
  • Example: small tight loop accessing huge array sequential

Case: Caching of Web Page

  • A web page is a file → fetch via http
  • Web access show temporal locality

History of Web Caching

  • early to mid 1990: caching of web pages by browser very successful
  • late 1990s: long slow decline in success of caching by browsers
  • today: very low browser cache hit ratio (20 -40%)

Web Monetization Neutered Caching

  • Advance knowledge of user intent & desires → money
      • inferring these from web accesses is a good heuristic
  • business want to know your every click
    • so every server marks pages as “uncacheable”
    • though the content is actually cacheable
  • Caching could be compatible with web monetization
    • browsers report all clicks
    • client can be trusted
  • client caching hides this knowledge from the server

Solution: Content Delivery Networks (CDNs)

Proxy caching that sells click insight
  • effectively third-party caching sites trusted by business
  • late binding of parasitic content by caching site
  • IC = Intrinsic Content (reachable)
  • PC = Parasitic Content (IC + PC is marked uncacheable)
