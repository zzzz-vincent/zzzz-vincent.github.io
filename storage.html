Intro: What is Storage Systems

Memory/storage hierarchy

Balancing performance with cost
  • Small memories are fast but expensive
  • Large memories are slow but cheap
Exploit locality to get the best of both worlds
  • locality = re-use/nearness of accesses
  • allows most accesses to use small, fast memory

Persistence

Storing data for lengthy periods of time
  • despite component failures, SW bugs/upgrades, human error
To be useful, it must also be possible to find it again later
  • this brings in data organization, consistency, and management issues
This is where the serious action is - and it does relate to the memory/storage hierarchy

Layers and Interfaces

Logical: Program ←→ File System ←→ Controller Firmware
Physical: Client ←→ Server ←→ I/O Controller ←→ Device ASIC
 

Solid-State Disks

  • Solid-state storage technology: Non-volatile memory chips with a disk interface
  • NAND Flash SSDs: SSD architecture and storage organization
  • SSD performance and the Flash Translation Layer (FTL):
    • Write amplification
    • Dynamic LBA-to-page mappings
    • Block cleaning
  • Media wearout
    • Wear leveling

SSD Technologies

  • RAMdisks: lots of DRAM together with a battery. Very expensive, very fast, limited poweroff data retention. But, coupled with a disk, battery can allow saving of all data
  • Flash: Flash != SSD, but most SSD is NAND flash.
    • NOR: lower density, word accessible, slow write, ~EEPROM
    • NAND: write in pages, erase in blocks of pagesEmbed a computer to manage wear/retention
  • Emerging technologies of note (a.k.a. NVM or SCM)
    • Currently cost as much or more than DRAM
    • Phase Change: amorphous vs lattice depend on heating
    • Magnetic RAM: magnetic bit in middle of DRAM cell

NAND Flash SSD is a little computer

  • Storage: flash chips
  • Access: multiple independent access channels
  • Interface: eg, SATA
  • Controller: computer + RAM
    • Processes cmds / Drives channels / Write behind / Allocation / Wear leveling
  • Flash Translation Layer FTL runs here

Storage Device Interface View:

  • Storage exposed as linear array of logical blocks (LBAs)
    • Addressed by “Logical Block Number” (LBN, a.k.a. LBA)
    • block sizes: maybe disk sector size (0.5-4KB), maybe memory or Flash page size (4KB), maybe bigger
    • Number of blocks: device capacity / block size

Performance

  • Single-page random reads
    • much, much lower average latency than mechanical disks
    • orders of magnitude higher throughput than mechanical disks
  • Peak bandwidth
    • depends on interface speed and chip parallelism
    • varies widely across products
  • Write performance is much more complicated
    • Single-page write is usually 100s of microseconds
    • write throughput more lower than read throughput due to write amplification

Problem: Write Amplification

Write amplification (WA) is an undesirable phenomenon associated with flash memory and solid-state drives (SSDs) where the actual amount of information physically written to the storage media is a multiple of the logical amount intended to be written.
How it works:
  • flash memory must be erased before it can be rewritten
  • erased granularity (usually block) > write granularity (usually page)
  • result in moving (or rewriting) user data and metadata more than once
How to reduce:
  • Amplification worst for small, random writes → freely remap between host & NAND addresses
    • Write LBAs to some place other than where they were before
    • Group bunch of different small writes into full blocks
      • Write at sequential write rates
    • Leaves “holes” in other blocks (where old data was)
      • At some point, “clean” out the holes by reading non-hole pages of a bunch of old blocks and writing back a smaller number of whole blocks
    • Rate of cleaning depends on amount of unallocated space
      • Over-provisioning
        • Controller reserves X% hidden space (ie. 10, 20, 50%) → this pushes up cost
        • Pre-clean/background clean lots of blocks (i.e., use idle time)
        • Leave partially deleted blocks longer to gather overwrites
      • Have host provide “delete” notifications (aka “TRIM”)
        • Write requests assign values (info) to locations (LBAs)
        • When traditional OS decides some info is no longer needed, it puts the assigned LBAs on a free list, but does not tell SSD
        • SSD can not free “deleted” info until eventual overwrite
        • Solution: OS can issue “TRIM” commands to free “deleted” info
        • By using TRIM, host reduces cleaning costs and need for over-provisioning

Performance implications of map size

  • Map granularity = flash block (256KB)
    • average I/O latency (for TPC-C workload) = 20 ms
  • Map granularity = flash page (4KB)
    • average I/O latency (for TPC-C workload) = 0.2 ms
    • 64 times larger map
    • Complexity of (background) cleaning

Wear leveling: spread writing evenly in SSD

  • Floating gate insulator degrades, accumulating charge that interferes, even when just reading. Stored charge leaks over (months to) years.
  • Blocks not written for a long time need to be rewritten → Read-erase-write or read-move/clean-erase
  • Each block can only survive a given number of erase/program cycles
    • 500 (USB stick), 3K (embedded), 10K-300K (enterprise)
  • Each block wears independently, so a heavily written block can wear out long before a mostly-read block
    • Wear leveling is remapping of addresses to better balance the number of erase/program cycles seen by each block

Wear leveling Benchmarks

  • With limited BW into SSD, lifetime is reasonable
    • N GB @ M MB/s = 1000*N/M secs to write all pages once
    • 128 GB, 100 MB/s, > 4 years 100% writing 24x7
  • Really need to include “write amplification” writes and wear leveling writes
    • Most sophisticated SSD controllers claim 10% extra
  • Can reduce “over-provisioning” as blocks “wear out”
    • Degrades performance, then stops offering full size

Increased Density through more bits per cell

  • Increases in density from two sources
    • Smaller feature size (Moore’s law at work)
    • More bits (more distinct drain current levels)
  • More bits per cell generally means lower cost-per-bit but also lower endurance and lower performance
    • multi-bit-per-cell Flash provides additional options for system designers
 

Hard Disk Drives

Disk

Service Time Components
Disk Structure
Tracks and Sectors
  • Addressable unit is a sector
  • Servo = bit pattern used for centering on track
  • Sector breaks down into several different fields
    • Typical data portion size – was 512-520 bytes, now 4096 bytes
    • Typical format:
      • sync followed by address field (cyl, head, sector, crc)
        • crc used to verify cyl, head, sector info
      • gap followed by the data
      • ecc over the data - verify data and correct bit errors
      • header, ECC and gaps typically use between 40 and 100 bytes
ECC (Error correction code)
  • At such high densities, problems occur frequently
  • ECC detects and can allow on-the-fly correction
  • Important consequence: when writing a sector, one ends up with one of three states. NEVER just partially modified (and able to pass ECC check)
    • all written
    • all not written
    • sector destroyed

Response time for disks

  • Response time (requester-visible service time): Queue time + Access time
  • Access time (service time for a disk access): Command + Seek + Rotation + Transfer

Seek Time:

  • Time required to move head over desired track
  • A seek has up to four components:
    • accelerate
    • coast at max velocity (only if going far enough to reach max velocity)
    • decelerate
    • settle onto correct track
      • even required for switching tracks (heads not always all aligned)
      • takes extra time to settle before writing
        • need to make extra certain to avoid destroying adjacent data
        • reads, on the other hand, can gamble
      • short seeks today just rely on the track settling mechanism

Rotational Latency

  • Time required for the first desired sector to reach head
  • Depends on rotation speed → measured in Rotations Per Minute (RPMs)
  • Computing average rotational latency
    • for almost all workloads, we can safely assume that there is an equal likelihood of landing on any sector of the track. This gives equal probability of each rotational latency. Thus, average rotational latency is time for 1/2 revolution.
    • eg. for 7200 RPM: one rotation = 60s / 7200 = 8.33 ms, average rotational latency = 4.16 ms

Media transfer time

  • Time for needed sectors to rotate under head
  • Computing transfer time
    • simple case: all on one track: sectors desired * time for revolution / sectors per track
    • more complex case: spread across two or more tracks
      • add a head switch time, as needed
      • add one or more track switch times, as needed

Modern disk performance characteristics

  • Seek times: 1 – 6 ms, depending on distance
    • Average 2 – 4 ms
    • Not necessarily improving
  • Rotation speeds: 4,200 – 15,000 RPM
    • average latency of 2-12 ms
    • Typically slowing down
  • Data rates: 60-125 MB/s*, depending on zone
    • avg sector xfer time of 25 us
    • improving at 0-20% per year

Classes of disk drives

  • High-capacity – being converted to SMR, MAMR drives
    • for disk-based backup, set top boxes, cloud storage
    • 16 TB today, slower speeds
  • High-speed – market being destroyed by NAND flash
    • for transaction processing
    • lower capacity, faster rotation, faster seeks
  • Enterprise quality – replace High-speed w/ “safer”
    • e.g., legacy private data center
    • legacy private data center
  • Desktop – market being destroyed by laptops & cloud
    • e.g., for cheap servers and PCs
    • medium capacity, medium speed, lower robustness

Benchmark

Zone

  • Outer tracks are longer than inner ones → hold more data
    • benefits: increased capacity and higher bandwidth
    • issues:
      • increased bookkeeping for LBN-to-physical mapping
      • more complex signal processing logic → because of variable bit rate timing
  • Zone: all tracks in each zone hold same number of sectors

Defect Management

  • Portions of media can become unusable
    • both before installation and during use
  • Need to set aside physical space as spares
    • simplicity dictates having no holes in LBN space
    • many different organizations of spare space
  • Two schemes for using spare space to handle defects
    • remapping: leave everything else alone and just remap the disturbed LBNs
    • slipping: change mapping to skip over defective regions
      • One spare sector per track
      LBN remapped from defective sector to spare
      LBN mapping slipped past defective sector

Computing physical location from LBN

  • check list of remapped LBNs → defect management
  • figure out which zone contains the LBN
    • can simply iterate through table of per-zone startlbn values
    • with knowledge of zone comes startlbn, firstcyl, and sectspertrack
  • lbn_zone = lbn - firstlbninzone
  • cylno = firstlbninzone + lbn_zone / sectspercyl
  • surfaceno = (lbn_zone % sectspercyl) / sectspertrack
  • sectno = lbn_zone % sectspertrack

File System Storage Layout

📚
Reading 1. File System Implementation, 2. Locality and The Fast File System

What Device Exposes (Device Interface):

  • Storage exposed as linear array of logical blocks (LBAs)
    • Addressed by “Logical Block Number” (LBN, a.k.a. LBA)
    • block sizes: maybe disk sector size (0.5-4KB), maybe memory or Flash page size (4KB), maybe bigger
    • Number of blocks: device capacity / block size

What the device does (Map LBNs to PBNs)

  • Generally done in device firmware today
    • HDD firmware maps LBNs to sectors
    • SSD firmware (FTL) maps LBNs to pages
    • Details usually hidden from external software (OS, FS, etc.)

What the user sees (Basic FS structures)

  • Directories are also files, and directory entries are their data
  • A common approach to recording a block list

Inode (index node)

  • inode stores metadata (except file name)
    • file size/device/access/ctime, mtime, atime/uid/gid/links
  • inode size: 128 bytes to 256 bytes
    • number of size are fixed in initilization
      • when init, disk has 1) data space and 2) inode table
    • each file needs one inode, cannot have more files if run out of inode
  • i-number: each inode has an i-number (low-level name), operation system uses it to identify different files
  • open a file: 1) get inode → 2) get inode info (pointers) → 3) get block for data
  • data access through multi-level index: through direct pointers, indirect pointers, double indirect pointer.
    • each inode has fixed number of direct pointers, one indirect pointer and one double indirect pointer.
    • direct pointer: stores disk addresses of data blocks
    • indirect pointer: points to a block contains more pointers. Assume 4KB block, and 4-byte disk address, then one direct pointer can point to 4KB * 1024 space
    • double indirect pointer: similar to indirect pointer, but size is 1024 * 1024 = 4GB
    • if still need more, triple indirect pointer can be added

File System Storage Layout

  • Components:
    • Data Region: user data storage
    • Inode table: inode for each file
    • Allocation Structures: track weather inodes/blocks are free
    • Superblock: information about the file systems (num of inodes, data blocks …)
  • Goal: map file system structures and data to LBNs
    • Problem 1: What’s free? Keep track of available LBNs
      • Usually at granularity of file system “blocks” (may span LBNs)
    • Problem 2: What’s next? Allocate a free block when needed
      • Performance (and other) implications when deciding which to pick

Solution 1 (Naive): Free List

  • Maintain linked list of free blocks → eg. use space in unused blocks to store pointers
  • Grab block from list when you need one → eg. treat list as a stack
  • Problem: rarely a performant solution

Solution 2 (Most Common): Bitmap

  • Large array of bits, one per allocatable unit
    • Two states per item: “free” or “in use”
  • Scan array for “free” bits when we need a block
    • Simple: take first “free”item in array
    • Better: look for “free” regions or for particular patterns

Solution 3 (Sometimes Better): Free Extent List

  • Don’t have to be a list
  • Extent: a contiguous range of free blocks
  • Maintain sorted list(s) of “free”extents
    • separate lists for tiny, medium, large extents
    • Merge freed, contiguous extents into larger ones
  • Pull space from a free extent, as needed
    • Accommodate misalignment or allow for room to grow in extent
  • Downside: fragmentation – many small unfilled extents
    • Will have to use them when free space is low (bad performance)

FS Storage Layout for HDDs

  • FS performance (largely) dictated by disk performance
    • Disk performance sensitive to seeking, rotational delays
    • We can control this: optimize FS block allocation algorithm
  • Two technology drivers yield two goals for HDDs
  • Locality: minimize seeking between requests
  • Transfer size: prefer fewer, larger transfers to amortize positioning costs
  • Locality (closeness)
    • reduce seek times by putting related things “close” to each other → close means nearby in LBN space
    • generally, benefits can be in the 2x range
  • Transfer size (large transfers)
    • amortize each positioning delay by accessing lots of data
      • means getting more per fetch == prefetching
    • generally, benefits can reach into the 10x range

Old Unix File System

  • A very simple data structure with super block ( w/ free-list of block), inodes, data
  • Poor performance
    • E gets spread across the disk. Then, when accessing E, you don’t get peak (sequential) performance from the disk

Fast File System (FFS)

  • Source of many still-popular optimizations
  • For locality (closeness) – cylinder groups
    • A single cylinder is a set of tracks on different surfaces of a hard drive that are the same distance from the center of the drive
      • They read or write together from all heads, with order from head 0 to head N.
      • Change of cylinder: mechanically, Change of head: electronically
    • called allocation groups/block groups in many modern FSs (e.g., ext*, xfs)
      • modern drives do not export enough information for the file system to truly understand whether a particular cylinder is in use, but a logical address space of blocks and hide details of their geometry from clients
      • cylinder groups == block groups from performance view
    • allocate inode in same cylinder group as directory
    • allocate first data blocks in same cylinder group as inode
  • For large transfers (amortization of positioning costs)
    • Larger FS block size: more data per disk read/write
      • Avoid internal fragmentation for small files by subdividing into “fragments”
    • Allocate contiguous blocks when possible
      • Start allocation search from last block #
    • Prefetching: fetch more data than requested
      • Pay positioning cost once, get multiple blocks
      • Read-ahead: kick in when sequential access is detected
      • Trade-off: disk may have already prefetched up to a few tracks, but a large request can delay normal requests
  • Small files is a problem
    • Performance may degrade over 50% with aging
    • Problem: small files lead to small I/Os
      • Cause: independence of files
      • Cause: levels of indirection (i.e., directory, inode, data, …)
      • Most files are small (e.g., 70% smaller than 8KB)
    • Solution: group small files + metadata for larger I/Os
      • co-locate related information and access together
      • over aggressiveness costs little

Log-structured File System (LFS)

  • All about amortization, try to achieve large writes
    • buffer and remap new versions of data into large segments
    • segment-sized writes make efficient use disk head
  • LFS vs. FFS
  • shadow paging
    • completely write a segment to replace several older blocks
    • “cleaning” reclaims space from fragmented segments
      • basic idea: condense still-live blocks from several current segments into fewer new segments, increasing number that are empty
      • this is the primary downside: may be expensive, that is, may cost more than benefit gained
  • Cleaning can require significant work
    • Of course, read and write of each live block
      • to move it from segment being cleaned
      • can just read entire segment, for amortization
    • Must first determine which blocks are live
      • requires knowing what each block is.
      • a “segment summary” structure embedded in each segment and can be cached or even pinned in RAM
    • requires knowing whether block is still live
      • can look at inode for data blocks and inode map for inodes
      • depending on cache state, can involve many reads
    • Good news: can be done in background / idle time
      • unless free space gets too low, then may be on-demand

FS Storage Layout for Flash SSDs

  • The mechanical performance drivers aren’t present
  • early SSDs did minimal remapping: LFS approaches resulted in much lower write amplification
  • most modern FTLs (flash translation layer) do LFS-like remapping into SSD blocks
    • overheads can be higher for small random writes: mapping table size, amplification, and cleaning/erase frequency
  • FTLs know about LBNs, not relationships among them
    • related data may be spread among SSD blocks, based on write timing
      • can help with parallelism for that data (if spread across planes)
      • can result in more cleaning overhead

F2FS: LFS-like FS specialized for Flash

  • Trick 1: set segment size = SSD block size
    • most writes fit and fill one SSD block
    • most SSD blocks 100% used or 0% used (in SSD’s view)
    • fine-grained mapping and cleaning only in F2FS → less FTL overhead
  • Trick 2: separate hot/cold data
    • put hot data in different segments than cold data
      • hot data more likely to be overwritten
      • cold data more likely to still be alive when cleaning
      • separating should reduce amount of data moved
    • benefits from knowing FS-level data relationships: inode in same segment with file data

File System

Application Interface

  • What apps see at the highest level (in most systems)
    • File contents: sequence of bytes
    • Basic operations: open, close, read, write
  • open starts a session and returns a handle
    • int fd = open(const char *path, int oflag, ...);
    • In POSIX (e.g., Linux and various UNIXes)
      • Handle is a process-specific integer called a file descriptor
      • Session remembers current offset into file
      • For local files, session also postpones full file deletion
  • read and write access bytes at some offset in file
    • ssize_t ret = write(int fd, void *buf, size_t nbyte);
    • Offset explicitly provided or remembered by session
  • close ends the session and destroys the handle
    • ssize_t ret = read(int fd, void *buf, size_t nbyte);
  • A few FSs treat open-to-close as atomic sessions. All or none of changes happen, isolated from other opens
    • ssize_t ret = close(int fd);

Accessing Open Files

  • Two opens of the same file yield independent sessions
  • A session can be shared across handles or processes

Passing data between kernel and application

  • Common approach: copy it
    • as in read(fd, buffer, size) or write(fd, buffer, size)
    • simplifies things in two ways
      • application knows it can use the memory immediately
        • and that the corresponding data is in that memory
      • kernel has no hidden coordination with application
        • e.g., later changes to buffer do not silently change file
  • Sometimes better approach: hand it off
    • as in char *buffer = mmap(addr, len, prot, flags, fd, off)
      • read/write avoids copying to/from user-space buffer
      • processes can share (but not private writable mappings) big files
    • Downsides
      • makes file caching more complex
      • can be confusing for application writers
      • sometimes not much performance improvement: fragmentation, messing with TLB (flush at unmap), faulting to load data in

Managing file data in the kernel: buffers

  • Usually used for two functions: staging area and cache
  • Two parts of each “buffer”
    • header describing controls and buffer containing data

In-memory Structures versus On-disk

  • The two do not have to match exactly → one just needs to be able to translate from one to the other
  • Good reason to be different
    • on-disk structures should be compact
    • in-memory structures should fit space/time trade-offs
    • on-disk structures should avoid pointers
    • in-memory structures may find pointers highly valuable

Supporting Multiple File System

  • can also support non-local FSs:

Vnode Layer

  • Problem: Want to be able to have multiple file systems at once
  • Solution: virtual file system layer
  • Everything in kernel interacts with FS via a virtualized layer of functions
  • TThese function calls are routed to the appropriate FS-specific implementations (once the correct FStype has been identified)

Example Vnode Operations

  • lookup(Vnode *dir, char *name, Vnode **result)
    • lookup name in directory dirp, returning child’s vnode
  • rdwr(Vnode *file, uint64_t off, uint32_t len, uio *uio)
    • read len bytes at offset off into user space at *uio
  • create(Vnode *dir, char *name, Attr *attrs, Vnode **filep)
    • create a file with the specified name, with the specified attributes.
  • remove(Vnode *dir, char *name)
    • delete a file
  • rename(Vnode *sdir, Vnode *ddir, char *sname, char *dname)
    • rename a file between one or two directories
  • getbuf(Vnode *file, uint64_t off, uint32_t len, buf **bp)
    • load a buffer with the contents of the file, returning a referenced buffer
  • cleanbuf(Vnode *file, buf *bp)
    • clean the specified buffer to the disk

Directory and File Structure

  • Directory
    • Directories to translate file names to inode IDs
  • Managing namespace: mount/unmount
    • One can have many FSs on many devices, but only one namespace
    • one must combine the FSs into one namespace
      • starts with a “root file system” → the one that has to be there when the system boots
      • “mount”operation attaches one FS into the namespace → at a specific point in the overall namespace
      • “unmount”detaches a previously-attached file system
  • Find root directory
    • Need enough information to find key structures
      • allocation structures
      • inode for root directory
      • any other defining information
    • Common approach: se predetermined locations within file system
    • Alternate approach: some external record

Different Kinds of Names

  • Direct versus indirect
    • hard link: name translates into object identity
      • unaffected by other names
      • not always possible (e.g., across file systems)
    • soft link: name translates into another name
      • affected by changes to the “another”name (e.g., dangling links)
  • Absolute versus relative
      • two hard links to a file
      symbolic link to a file
    • absolute: name is self-contained
      • e.g., /usr/local/bin/ssh as opposed to ./ssh
      • Must be explicitly updated when moving portions of namespace
    • relative: name is context-dependent
      • e.g., relative to current directory name such as ../foo/bar
      • can lead to ambiguity due to absence of closure

Walking Through an open Operation

  • int fd = open(“/foo/bar”, RO);
      1. translate file name to inode identifier
        1. search root directory for “foo”using vnode_lookup
        2. search directory “foo” for “bar” using vnode_lookup
        3. use Directory Name Lookup Cache (DNLC) first for each lookup step!
        4. check access rights at each step
      1. create a vnode structure for inode
        1. lookup inode in inode cache; fetch from disk if necessary
        2. initialize vnode structure appropriately
      1. create open file structure
        1. initialize, pointing to new vnode
      1. fill in fd table entry
        1. pick unused entry in table; have it point to new open file structure
      1. return corresponding index into fd table

Walking Through an read Operation

  • int retcode = read(fd, buffer, size);
      1. index into fd table to get open file object
        1. get vnode from open file object
      1. call vnode_read(vnode, offset, buffer, size)
        1. usually calls back to generic read_write code
      1. find vnode’s buffer containing offset in buffer cache
      1. if buffer is invalid, vnode_getbuf fills it
        1. may read and interpret indirect blocks
        2. requests device driver to fill buffer with data
      1. copy data from cached buffer to user space buffer
        1. and advance offset past data copied in file object
      1. repeat last three steps until size reached
      1. return to application
        1. update open file object’s offset on the way

Walking Through an write Operation

  • int retcode = write(fd, buffer, size);
      1. index into fd table to get open file object
        1. get vnode from open file object
      1. call vnode_write(vnode, offset, buffer, size)
        1. usually calls back to generic read_write code
      1. find vnode’s buffer containing offset in buffer cache
      1. if buffer is invalid, vnode_getbuf fills it
        1. or marks it valid if entire buffer will be overwritten
        2. may interpret and create indirect blocks
      1. copy data from user space to cache buffer
        1. and advance offset past data copied in file object + mark buffer dirty
      1. repeat last three steps until size reached
      1. return to application
        1. leaves buffers as dirty in buffer cache
